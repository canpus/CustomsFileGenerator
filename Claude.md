# CLAUDE.md — AI 编程工作最高准则(根总纲)

> **地位**:本文档为所有项目的 AI 编程最高准则.项目级 `CLAUDE_*.md` 可补充项目特有规则,但不得与根总纲冲突.
> **版本**:4.0.0
> **最后更新**:2026-05-10
> **目标用户**:有 Python 基础的复学者

---

## ⚡ 硬约束卡(本条优先级最高,与其他章冲突时以本条为准)

1. 输出代码可直接保存并运行(含所有 `import` + `if __name__ == "__main__"` 入口)
2. 所有可能失败的操作(I/O,网络,数据库,类型转换)必须包裹异常处理,except 块含中文错误说明 + 原因 + 解决方法
3. API Key,密码,Token 一律走 `.env`,禁止硬编码
4. 不确定的业务决策标注 `TODO:待确认`,保持可配置,禁止自行假设
5. 不得擅自修改用户的配置文件(`.env`,`settings.json`,`package.json` 等)
6. 耗时超过 2 秒的操作必须打印实时进度
7. 函数默认不得修改通过参数传入的可变对象(字典,列表等),必须基于传入数据生成并返回新对象;如确需原地修改,必须显式标注副作用.嵌套数据结构复制须使用 `copy.deepcopy()`
8. 非 Optional 的关键函数入参在入口处必须做真值断言,为 `None` 时立即抛出含业务含义的异常,禁止静默向下传递

## 三,AI 行为总纲

1. **"代码输出即能运行"是第一铁律.**
2. 不确定的业务决策 → 标注 `TODO:待确认`,保持可配置,**禁止自行假设**.
3. 不得擅自修改配置文件(`.env`,`settings.json`,`package.json` 等),修改前须征得同意.
4. 不得删除或覆盖用户已有代码,除非用户明确指示.
5. 每次代码输出后,附带简要说明:做了什么,为什么这样做,接下来做什么.

## 四,工程质量 `[硬约束]`

> 本章为 AI 输出代码的最低质量门槛.当其他规则与本章冲突时,以本章为准.

### 4.1 零报错原则

所有可能失败的操作必须包裹异常处理,except 块中必须打印中文错误说明 + 原因 + 解决方法.禁止使用不确定存在的默认值(系统路径,环境变量,硬编码绝对路径).具体 Python 代码示例见 `Claude.python.md`.

### 4.2 进度可见原则

耗时超过 2 秒的操作必须实时打印进度.程序启动时打印关键配置,完成时打印输出文件列表.代码示例见 `Claude.python.md`.

### 4.3 错误自解释原则

错误信息必须严格包含以下三要素, 并统一使用标准方括号标签输出:
- `[错误]`：发生了什么 (中文描述)
- `[原因]`：可能的原因 (1-2 条)
- `[排查]`：解决方法 (具体步骤)

### 4.4 完整交付原则

输出代码必须可直接保存并运行,包含所有 import 语句,包含 `if __name__ == "__main__"` 入口块,附带运行说明注释块(依赖安装 + 运行命令 + 预期输出).

### 4.5 环境自检原则

程序入口必须先做运行环境检查,检查不通过则打印中文修复指导并退出.代码示例见 `Claude.python.md`.

### 4.6 降级策略原则

高级功能不可用时自动降级而非报错退出:GPU 不可用切 CPU,LLM API 不可用切本地缓存,可选依赖未安装则禁用相关功能并打印提示.

## 五,AI 协作规范

### 5.1 开发流程

1. **理解需求**:读取 `plan.md` 和相关文档,确认当前阶段
2. **声明任务**:说明做什么,涉及哪些文件,属于哪个阶段
3. **编写代码**:逐模块实现,控制在行数上限内
4. **附带说明**:输出后简要说明做了什么,为什么这样做
5. **提醒审核**:每完成一个模块,主动提醒用户审核
6. **更新文档**: 在当前项目目录下维护 `README.md`. 每次完成 `plan.md` 中定义的阶段性任务或涉及关键配置变更时, 必须在 `README.md` 的 "修订历史" 章节中追加记录 (含版本号, 日期及简短变更说明).
7. **日志系统**: 统一在项目根目录的 `logs/` 文件夹下实现双文件滚动日志系统 (使用 `logging.handlers.RotatingFileHandler`, 单文件上限 5MB, 备份数 3 个, 编码统一为 UTF-8):
    - `operations.log`: 仅记录用户显式操作 (如点击按钮, 切换页面) 与核心处理进度. 级别限制为仅 `INFO` (禁止写入 WARNING 及以上日志).
    - `errors.log`: 记录所有捕获的异常, 必须严格包含堆栈信息 (exc_info). 且写入的日志文本必须严格嵌套 `[错误]`、`[原因]`、`[排查]` 三要素标牌, 禁止使用任何 Emoji 符号. 级别为 `WARNING` 及以上.

### 5.2 开发纪律

- 每次只实现当前计划阶段的子任务,跨阶段开发须用户明确允许
- 模板文件只读不写,修改复制到新文件中操作
- 新依赖安装前须征得用户同意,先更新依赖文件再执行安装
- 每个代码块包含完整函数体,完整 import,完整 `if __name__ == "__main__"` 入口
- 代码示例保留完整逻辑,无 `...` 或 `# 省略` 截断

> 交付物自检清单已移至 `Claude.python.md` 供 Python 项目专用.`[建议]` 用户审核清单请参见根 `Claude.md` 附录.

## 六,安全基线

1. API Key,密码,Token **一律走 `.env`**,严禁硬编码
2. `.env` 绝不提交 Git,提供 `.env.example` 模板
3. 数据库操作使用**参数化查询**,禁止拼接 SQL 字符串
4. 日志中不得输出密码,API Key,用户隐私数据
5. 所有用户输入必须在**后端校验一次**(类型,范围,枚举值)

# CLAUDE.md — 全局 AI 编码规范

## P0. 最高优先级

1. **先理解再修改**
   - 先读取 `plan.md`、`README.md`、项目现有结构与相关代码。
   - 只实现当前阶段任务；跨阶段开发必须先说明。

2. **不擅自做业务决策**
   - 需求、字段含义、边界规则、格式约束不明确时，保留可配置项并标注 `TODO: 待确认`。
   - 禁止把猜测伪装成既定事实。

3. **不擅自改用户关键配置**
   - 禁止未经说明直接修改 `.env`、`settings.json`、`package.json`、数据库结构、部署配置。
   - 涉及新增依赖、迁移、批量重构时，先明确影响面。

4. **安全优先**
   - 密钥、口令、Token 一律从环境变量或配置文件读取，禁止硬编码。
   - `.env` 不入库，提供 `.env.example`。
   - 日志、报错、示例数据中禁止泄露密钥、隐私、生产路径。

5. **Python 环境铁律**
   - 禁止使用全局环境、系统基础环境或 base 环境。
   - 必须在**项目根目录**创建、激活并使用 `.venv` 虚拟环境。
   - 所有 Python 依赖安装、脚本运行、测试执行、格式化、Lint、构建，均必须在该 `.venv` 中完成。

6. **代码改动必须最小且可解释**
   - 优先修根因，避免表面补丁。
   - 不删除或覆盖用户已有逻辑，除非任务明确要求。
   - 每次输出都要说明：**改了什么、为什么这样改、还剩什么风险**。

7. **输出必须可落地**
   - 给出的代码、命令、路径、配置应可直接执行或直接保存。
   - 禁止用 `...`、伪代码、半截函数体代替正式实现，除非用户明确只要思路。

## P1. 默认工程规则

### 1. 设计与结构

- 优先 **高内聚、低耦合、单一职责**。
- 优先改小范围文件；避免无关重命名与大面积格式噪声。
- **单文件代码量目标**：`<= 300` 行；`300~500` 行需检查是否存在自然拆分点；`> 500` 行应说明理由或拆分。
- **圈复杂度目标**：`<= 10`；超过时优先拆函数、拆分支、提取策略。
- 禁止循环导入。

### 2. Python 风格

- 遵循 **PEP 8**。
- 字符串默认使用双引号。
- 优先使用 `pathlib.Path`、`f-string`、上下文管理器、标准库。
- 禁止 `from x import *`。
- 非必要不要在函数内部导入；如为解除循环依赖或降低启动成本，需有明确理由。

### 3. 类型与接口

- **公共函数、公共方法、核心数据结构** 必须有类型注解。
- **公共函数** 必须有 docstring；私有且语义明显的短函数可省略。
- 函数命名应体现动作或结果，禁止含糊命名。
- 配置、请求、响应、领域实体优先使用显式模型；是否使用 `Pydantic` 取决于项目栈，不强制一刀切。

### 4. 错误处理

- **I/O、网络、数据库、子进程、反序列化、外部 API** 等边界操作必须显式处理异常。
- 禁止静默吞异常；至少记录上下文或转换成更清晰的业务错误。
- **内部纯计算函数** 不强制层层 `try/except`；允许自然抛错，由边界层统一处理。
- 用户可见错误信息使用中文；日志保留技术细节。

### 5. 数据与副作用

- 默认不要修改传入的可变对象；优先返回新对象。
- 如确需原地修改，函数名、docstring 或注释里明确标注副作用。
- 避免可变默认参数，使用 `None` 作为哨兵值。
- 嵌套结构复制时，根据实际需要选择浅拷贝或深拷贝；**不要机械地一律 `deepcopy()`**。

### 6. 校验与健壮性

- 对关键输入做 **语义校验**，但不要把合法的 `0`、`False`、空字符串误判为非法。
- 对必须非空的参数，优先显式判断 `is None`、类型、范围、枚举值，而不是使用笼统真值断言。
- 对长耗时步骤、批处理、用户可感知等待，提供阶段性进度或状态反馈。

### 7. 配置与依赖

- 路径、URL、端口、密钥、开关等必须配置化。
- 维护 `requirements.txt` 或 `pyproject.toml`。
- 生产依赖建议锁定版本；开发期可按团队策略选择精确锁定或区间锁定。
- 新增依赖前先更新依赖文件，并说明用途。
- **Python 项目必须使用项目根目录 `.venv` 虚拟环境**，禁止使用全局或 base 环境。

### 8. 日志

- 统一使用 `logging`。
- 记录关键节点、异常上下文、必要输入摘要；避免记录敏感值。
- 如项目需要双日志文件，可采用：
  - `operations.log`：业务进度与用户操作
  - `errors.log`：异常与堆栈
- 日志方案应按项目需要启用，不强制所有脚本一开始就接入完整滚动日志架构。

### 9. 测试

- 修改业务逻辑、解析逻辑、数据转换逻辑时，应补或改相应测试。
- 单元测试优先覆盖：
  - 正常路径
  - 边界条件
  - 预期失败
- 测试不得依赖外部在线服务；LLM 调用必须 mock。

## P2. Python 专项规则

### 1. 常见陷阱

- 禁止可变默认参数。
- 循环内创建闭包时，显式捕获循环变量。
- 禁止共享全局可变状态承载并发结果。

### 2. 并发

- 禁止无限制并发。
- 使用 `Semaphore`、任务池或队列控制并发度。
- 单任务失败不得拖垮整个批次；汇总成功/失败数量。
- 复杂并发逻辑单独封装，不把并发细节散落在业务代码中。

### 3. I/O 与资源

- 大文件读取使用流式或分块方式，不一次性载入超大内容。
- 临时目录、文件句柄、数据库连接、网络连接必须确保释放。
- CLI 或批处理入口在必要时做环境自检；**不是每个脚本都强制带完整自检模板**。

### 4. 数据库与持久化

- 使用参数化查询，禁止拼接 SQL。
- Repository / DAO / Service 分层按项目复杂度决定；小项目避免过度设计，大项目避免 SQL 散落业务层。

## P3. AI 执行方式

1. **先给计划，再动手**
   - 说明本次要改哪些文件、对应哪个阶段、预期产物是什么。

2. **一次只解决一个清晰问题**
   - 避免把“修 Bug、重构、换架构、补测试、改文档”一次性混成一个无法审查的大提交。

3. **先局部验证，再扩大范围**
   - 优先运行最小必要测试、最短路径验证；再决定是否扩大验证范围。

4. **说明风险而不是隐藏风险**
   - 若无法确认真实运行结果，明确说明缺少哪些日志、样例、环境或接口信息。

5. **文档同步**
   - 影响使用方式、配置方式、目录结构、接口契约时，更新 `README.md`。
   - 若项目已有阶段计划文件，完成阶段任务后同步更新进度。

#### P01 — PackingGenerator 未继承 BaseGenerator

**位置**: [packing_generator.py](src/generators/packing_generator.py#L135) vs [base_generator.py](src/generators/base_generator.py#L26)

**描述**: `PackingGenerator` 在阶段 4 编写，`BaseGenerator` 在阶段 5 抽象。导致 PackingGenerator 复制了 ~180 行与 BaseGenerator 几乎相同的代码（沙箱管理、进度回调、输出路径解析、generate 流程）。如果 BaseGenerator 的流程被修改，PackingGenerator 不会受益。

**违反规则**: Claude.python.md 2.1 "模块化与文件组织" — 代码复用原则

**修复方向**: 让 `PackingGenerator(BaseGenerator)` 继承基类，删除重复代码。

#### P02 — num2words 依赖未列入 requirements.txt

**位置**: [invoice_generator.py](src/generators/invoice_generator.py#L400) 导入 `num2words`，但 [requirements.txt](requirements.txt) 中无此项

**描述**: `num2words` 被用于将金额转换为英文大写。虽然代码中有 `ImportError` 降级处理，但这是一个运行时必需依赖，不在依赖清单中会导致新环境部署时功能降级。

**违反规则**: Claude.md 硬约束卡第 1 条 "输出代码可直接保存并运行"；Claude.python.md 2.4 "依赖版本必须锁定"

**修复方向**: 在 `requirements.txt` 中添加 `num2words==0.5.14`。

#### P03 — num2words ImportError 静默吞掉

**位置**: [invoice_generator.py](src/generators/invoice_generator.py#L419-L431)

**描述**: `except ImportError` 返回数字格式降级（仅写 logger.warning），`except Exception` 虽记录了 error 日志但仍返回降级值。调用方无法区分"正常降级"和"转换失败"。

**违反规则**: Claude.python.md 2.3 "异常不得静默吞掉"

**修复方向**: 在生成上下文中，当 ImportError 发生时应在 orchestrator 层面标记 warning；当转换逻辑异常时应抛出明确错误而非返回降级字符串。

#### P04 — 模块内 try/except 裸吞异常

**位置**:
- [base_generator.py](src/generators/base_generator.py#L176-L179)
- [packing_generator.py](src/generators/packing_generator.py#L266-L271)

```python
except Exception:
    try:
        wb.close()
    except Exception:
        pass  # ← 完全静默，无日志
    raise
```

**描述**: 内层 `wb.close()` 失败时的异常被完全吞掉，没有日志。虽然外层的原始异常会被 re-raise，但 `wb.close()` 失败本身可能意味着资源泄漏。

**违反规则**: Claude.python.md 2.3 "异常不得静默吞掉"

**修复方向**: 内层 catch 至少记录 `logger.warning`。

#### P09 — 模块级可变全局单例

**位置**: [orchestrator.py](src/generators/orchestrator.py#L328-L340)

```python
_DEFAULT_ORCHESTRATOR: Orchestrator | None = None

def get_orchestrator() -> Orchestrator:
    global _DEFAULT_ORCHESTRATOR
    if _DEFAULT_ORCHESTRATOR is None:
        _DEFAULT_ORCHESTRATOR = Orchestrator()
    return _DEFAULT_ORCHESTRATOR
```

**描述**: 模块级可变单例存在风险——多线程首次调用 `get_orchestrator()` 存在竞态条件（虽然当前单线程 GUI 下不会触发，但设计上不规范）。此外 `_DEFAULT_ORCHESTRATOR` 被 `global` 重新赋值，属于 mutable global state。

**修复方向**: 使用 `threading.Lock` 保护，或直接用类属性 `Orchestrator._instance`。

#### P10 — SQL 字符串插值（即使有 allowlist 保护）

**位置**: [repository.py](src/db/repository.py#L156-L163)

```python
set_clause = ", ".join(f"{k} = ?" for k in updates)
cursor = conn.execute(
```

**违反规则**: Claude.md 安全基线第 3 条 "禁止拼接 SQL 字符串"

**修复方向**: 可维持现状（因为有 allowlist 保护），但建议在注释中明确标注安全边界。

#### P12 — `os.startfile()` 仅适用于 Windows

**位置**: [generate_page.py](src/gui/pages/generate_page.py#L443)

```python
os.startfile(str(zip_path.parent))
```

**描述**: `os.startfile` 是 Windows 专有 API，在 macOS/Linux 上会抛出 `AttributeError`。

**修复方向**:
```python
import sys
import subprocess
if sys.platform == "win32":
    os.startfile(str(path))
elif sys.platform == "darwin":
    subprocess.Popen(["open", str(path)])
else:
    subprocess.Popen(["xdg-open", str(path)])
```

## 四、其他建议（非必须）

1. **恢复测试** — 你提到已删除测试代码。建议在重构完成后，为每个改动过的模块编写至少 1 个冒烟测试（正常输入 → 预期输出）。无测试覆盖的重构风险极高。

2. **缺失 `plan.md`** — CLAUDE.md 多处引用 `plan.md`，但项目实际使用的是 `plan_v6.md`。建议创建 `plan.md` 作为活动计划的符号链接（或指向当前活跃版本）。

3. **Stage 6 恢复清单** — 代码中有 8 处 `[待迁移]` 注释标记了 Stage 6（DOCX 报关单）恢复时需要修改的位置。建议在 `plan_v6_a.md` 的附录 C 中维护一个 checklist，逐项跟踪。

4. **GUI 页面"客户管理"和"产品库"是占位实现** — [app.py](src/gui/app.py#L326-L333) 中 `customer` 页面复用 `OrderInfoPage`，`product` 页面复用 `TreeEditorPage`。在真正的客户/产品管理功能实现前，建议在 UI 上明确标注"功能开发中"。

5. **模块 `__init__.py` 可添加 `__all__`** — 当前所有 `__init__.py` 几乎为空，添加 `__all__` 有利于控制导出接口和 IDE 自动补全。
